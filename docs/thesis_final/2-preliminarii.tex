\chapter{Preliminarii}
\label{cap:preliminarii}

În acest capitol prezint noțiunile tehnice pe care se sprijină proiectul. Le includ aici nu ca o
listă de tehnologii, ci ca fundal pentru deciziile din implementare: de ce PostgreSQL rămâne sursa de
adevăr, de ce istoricul se păstrează prin instantanee, de ce accesul trebuie verificat pe fiecare
resursă și de ce căutarea este tratată ca model de citire separat.

\section{Aplicații web și sursa de adevăr}

Platforma este o aplicație web cu frontend React și backend Spring Boot. Frontend-ul trimite cereri
JSON către un API REST, iar backend-ul aplică regulile de domeniu și persistă datele. Într-un astfel
sistem, baza de date relațională nu este doar un loc de stocare, ci și componenta care poate garanta
integritatea prin chei externe, constrângeri și tranzacții.

În proiect, PostgreSQL este sursa unică de adevăr. Această alegere este importantă deoarece aplicația
are date care nu trebuie pierdute sau interpretate diferit: sesiuni de antrenament, token-uri de
reîmprospătare, relații antrenor-client, voturi și salvări în marketplace. Hibernate este folosit
pentru maparea obiect-relațională, dar schema nu este generată automat de aplicație. Ea este creată și
modificată numai prin migrații Flyway, iar configurația Hibernate este setată pe \code{ddl-auto:
validate}. Astfel, aplicația verifică schema la pornire, dar nu o schimbă implicit.

\section{Date de planificare și date istorice}

O diferență esențială în domeniul aplicației este cea dintre date de planificare și date istorice.
Datele de planificare descriu intenția utilizatorului: programe, zile, exerciții planificate, rutine,
săli și echipamente. Ele trebuie să rămână editabile. Datele istorice descriu însă ce s-a întâmplat la
un moment concret și nu ar trebui să se schimbe retroactiv.

Soluția folosită este modelarea prin instantanee. Un instantaneu nu înseamnă o copie a întregii baze
de date, ci copierea atributelor relevante într-o entitate istorică. De exemplu, o sesiune nu se
bazează pe numele curent al exercițiului, ci păstrează numele exercițiului copiat atunci când sesiunea
a fost pornită. Același principiu se aplică grupelor musculare, programului, zilei, rutinei, sălii și
echipamentului. Cheile externe către sursă pot exista în continuare pentru trasabilitate, dar trebuie
să poată deveni \code{NULL} fără a strica istoricul.

\section{Autentificare, autorizare și IDOR}

Autentificarea stabilește identitatea utilizatorului care trimite o cerere. În aplicațiile moderne,
un mod uzual de lucru este folosirea token-urilor JWT \cite{rfc7519}: serverul semnează un token de
acces, iar clientul îl trimite în antetul \code{Authorization}. Token-ul de acces trebuie să aibă o
durată scurtă, deoarece este prezent la fiecare cerere. Pentru sesiuni mai lungi se folosește un token
de reîmprospătare, care trebuie protejat mai strict.

Autorizarea răspunde la o altă întrebare: utilizatorul autentificat are voie să acceseze resursa
cerută? Într-un sistem multi-user, o greșeală frecventă este IDOR (\emph{Insecure Direct Object
Reference}), adică situația în care un utilizator schimbă un identificator într-o cerere și obține
datele altcuiva. O protecție corectă nu se bazează pe identificatori primiți de la frontend, ci pe
utilizatorul curent extras din token și pe interogări care includ proprietarul resursei.

Pe lângă regulile obișnuite pe roluri (RBAC), modul de coaching cere și reguli bazate pe relații
(ReBAC). Un utilizator cu rol de antrenor nu ar trebui să poată vedea orice utilizator, ci doar
clienții cu care are o relație activă. De aceea, rolul este doar primul filtru; al doilea este
relația antrenor-client acceptată.

\section{Căutare full-text și modele de citire}

PostgreSQL poate face căutare full-text prin \code{tsvector}, indecși GIN și funcții de ranking
\cite{postgresFTS}. Pentru colecții mici sau pentru interogări simple, aceasta poate fi o soluție
suficientă și mai ușor de operat decât un motor separat. Când apar însă cerințe precum toleranță la
greșeli de tastare, sinonime, scoruri de relevanță pe mai multe câmpuri, evidențierea fragmentelor și
fațete, un motor dedicat precum OpenSearch \cite{opensearch} devine mai potrivit.

În această lucrare, OpenSearch nu înlocuiește PostgreSQL. El este un model de citire derivat:
documentele indexate sunt reconstruibile din baza de date relațională, iar rezultatele sunt validate
din nou în PostgreSQL înainte de a fi returnate. Această abordare introduce consistență eventuală,
dar păstrează o sursă unică de adevăr și permite refacerea indexului în caz de eroare.

\section{Analiză deterministă a programelor}

Aplicația include și un analizor al structurii programelor. Scopul lui nu este să ofere recomandări
medicale și nici să înlocuiască un antrenor, ci să dea un feedback explicabil despre structură:
volumul săptămânal, acoperirea grupelor musculare, frecvența, echilibrul între regiuni, lungimea
sesiunilor și pauzele. Regulile sunt inspirate din literatura despre volumul antrenamentului
\cite{schoenfeld2017dose} și din formule simple pentru estimarea progresului, cum este formula lui
Epley pentru 1RM \cite{epley1985}.

Am ales o abordare deterministă, bazată pe scăderi fixe dintr-un scor, deoarece este mai ușor de
explicat și de testat decât o soluție de tip „AI coach”. Pentru o lucrare de licență și pentru un
produs care atinge antrenamentul fizic, această onestitate este importantă: sistemul poate semnala
probleme de structură, dar nu poate cunoaște recuperarea, tehnica, somnul, alimentația sau nivelul
real de efort al utilizatorului.
