Constrângerile importante includ: valori admise pentru enumerări, \code{days\_per\_week BETWEEN 1 AND 7},
\code{rpe BETWEEN 1.0 AND 10.0}, \code{finished\_at >= started\_at}, contoare nenegative, interdicția
ca un utilizator să fie propriul antrenor, publicarea doar cu \code{published\_at} nenul și indecși
unici parțiali pentru resursele șterse logic.

\begin{lstlisting}[language=SQL]
CREATE UNIQUE INDEX ux_workout_sessions_one_active_per_user
  ON workout_sessions (user_id)
  WHERE status = 'IN_PROGRESS';

CREATE UNIQUE INDEX gyms_user_name_active_idx
  ON gyms (user_id, lower(name))
  WHERE deleted_at IS NULL;

ALTER TABLE workout_templates
  ADD CONSTRAINT check_template_visibility_published
  CHECK (
    (visibility = 'PUBLIC' AND published_at IS NOT NULL)
    OR visibility = 'PRIVATE'
  );
\end{lstlisting}

\anexachapter{Listări de cod reprezentative}

Următoarele fragmente sunt scurtate pentru lizibilitate, dar păstrează forma regulilor implementate în
proiect.

\noindent\textbf{Listarea 1.} Analizorul de programe folosește reguli deterministe: fiecare sub-scor
pornește de la valoarea maximă și scade penalizări fixe pentru problemele găsite.

\begin{lstlisting}[language=Java]
private int scoreVolumeAndCoverage() {
    int score = 35;

    if (!lowerBody) {
        add("NO_LOWER_BODY", HIGH, "No lower-body work is planned.");
        score -= 12;
    }
    if (!pulling) {
        add("NO_PULL", HIGH, "No pulling movement is planned.");
        score -= 12;
    }
    if (!pushing) {
        add("NO_PUSH", HIGH, "No pushing movement is planned.");
        score -= 12;
    }

    for (String bucket : ALL_BUCKETS) {
        double volume = weeklyWeightedSets(bucket);
        if (volume <= 3) {
            add("MUSCLE_VOLUME_VERY_LOW", MEDIUM, bucket);
            score -= 3;
        } else if (volume > 25) {
            add("MUSCLE_VOLUME_EXCESSIVE", HIGH, bucket);
            score -= 5;
        } else if (volume > 20) {
            add("MUSCLE_VOLUME_HIGH", MEDIUM, bucket);
            score -= 2;
        }
    }

    return Math.max(0, score);
}
\end{lstlisting}

\noindent\textbf{Listarea 3.} Tabela \code{outbox\_events} există încă din prima migrare și poate fi
folosită ulterior pentru indexare tranzacțională.

\begin{lstlisting}[language=SQL]
CREATE TABLE outbox_events (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  aggregate_type  varchar(100) NOT NULL,
  aggregate_id    uuid NOT NULL,
  event_type      varchar(100) NOT NULL,
  payload         jsonb NOT NULL DEFAULT '{}'::jsonb,
  status          varchar(30) NOT NULL DEFAULT 'PENDING',
  attempts        int NOT NULL DEFAULT 0,
  available_at    timestamptz NOT NULL DEFAULT now(),
  processed_at    timestamptz,
  created_at      timestamptz NOT NULL DEFAULT now(),
  last_error      text,
  CONSTRAINT check_outbox_status
    CHECK (status IN ('PENDING','PROCESSING','PROCESSED','FAILED'))
);

CREATE INDEX outbox_events_processing_idx
  ON outbox_events (status, available_at, created_at);
\end{lstlisting}

\anexachapter{Configurație și rulare}

Fișierul \code{docker-compose.yml} pornește PostgreSQL 16, OpenSearch 2.13.0 și aplicația Spring Boot.
Imaginea aplicației construiește frontend-ul Vite în timpul build-ului Maven și îl include în jar-ul
Spring Boot, astfel încât interfața și API-ul sunt servite de aceeași aplicație la
\code{http://localhost:8080}.

\begin{lstlisting}
docker compose up --build
\end{lstlisting}

Configurația principală păstrează schema sub controlul Flyway și dezactivează Hibernate DDL. Modulul de
căutare este dezactivat implicit și este activat în compose prin \code{APP\_SEARCH\_ENABLED=true}.

\begin{lstlisting}
spring:
  jpa:
    hibernate:
      ddl-auto: validate
  flyway:
    enabled: true

app:
  security:
    jwt:
      issuer: workout-thesis
      access-ttl: 15m
      refresh-ttl: 14d
  search:
    enabled: ${APP_SEARCH_ENABLED:false}
    uri: ${APP_SEARCH_URI:http://localhost:9200}
    templates-index: templates_v1
    templates-alias: templates
    sessions-index: workout_sessions_v1
    sessions-alias: workout_sessions
\end{lstlisting}

Pentru dezvoltare, frontend-ul poate rula separat prin Vite pe portul 5173, iar backend-ul pe portul
8080. Testele de integrare folosesc Testcontainers și au nevoie de Docker funcțional, deoarece pornesc
PostgreSQL 16 și, pentru suita de căutare, un container OpenSearch 2.x.
